3 Arquitecturas para Offline-First

Elige la que mejor se adapta a tu producto SaaS

A

Dexie.js + Cola de Sincronización

Recomendada

Cómo funciona
  • Catálogo completo en IndexedDB (Dexie)
  • Ventas se guardan local primero
  • Al reconectar → sync automático
  • Service Worker detecta conexión
Características
  • Sin costo extra por cliente
  • Funciona en cualquier navegador
  • Instalable como PWA
  • Compatible con multi-sucursal

Ventajas

  • Madurez probada en producción
  • Bundle pequeño (~47kb)
  • API simple, fácil de mantener

Limitaciones

  • Stock "optimista" offline
  • Sync manual a diseñar
B

PGlite (SQLite en WASM)

Base de datos SQL completa en el navegador

Cómo funciona
  • SQLite real en el navegador
  • Mismas queries que Supabase
  • Sync manual con Postgres
  • Requiere WASM habilitado
Características
  • SQL nativo completo
  • Triggers locales posibles
  • Sin Supabase Pro requerido
  • Tecnología más nueva

Ventajas

  • Más potente a largo plazo
  • SQL idéntico local/servidor

Limitaciones

  • Bundle pesado (~3MB WASM)
  • Más complejo de implementar
  • Menos battle-tested
C

Service Worker Puro

Sin base de datos local — solo caché de red

Cómo funciona
  • Cache API responses en disco
  • Writes en cola simple
  • Sin IndexedDB estructurado
  • Background Sync API
Características
  • Menos código a mantener
  • Funciona con el código actual
  • Limitado en capacidad
  • Frágil con catálogos grandes

Ventajas

  • Implementación más rápida
  • Sin dependencias nuevas

Limitaciones

  • No escala para catálogos grandes
  • Búsqueda offline muy limitada
  • Menos robusto
Mi recomendación: Opción A — Dexie.js. Es la más madura, más ligera y la más fácil de mantener como producto SaaS. Haz clic en ella y dime si estás de acuerdo.
